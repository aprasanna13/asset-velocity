Design Document:
Global Search Subsystem

A unified context-aware navigation and query overlay optimized for pipeline operators. Prioritizes asset lookups in real-time without breaking situational awareness in high-stress triage environments.

Status Verified & Clean
Target Version v1.4.0 Release
Tech Stack React 18 • Fuse.js
Dependencies None (Statically Shared)
01 // PROBLEM & SOLUTION

The Business Problem

The Field Operations application serves operators managing complex, high-stakes industrial environments (such as pipeline networks and SCADA telemetry). These operators are responsible for monitoring live data, responding to critical anomalies, and executing maintenance workflows.

The Friction Cognitive Load

As system complexity grows, information becomes siloed across specialized views (Network Map, Parts Tables, Execution Console). Navigating away from SCADA triage to search for a part forces split-second context loss, which can be dangerous in industrial operations.

The Solution Unified Overlay

A context-aware global search bar in the application header. By dynamically prioritizing matches based on the active screen, and opening results in non-disruptive overlay drawers or map highlights, operators retain full situational awareness.

02 // COMPONENT MAP

Architecture & Component Flow

The global search feature acts as an elegant, low-risk "lens" over the existing application. It reads data flowing through the app and presents it in a unified, debounced results overlay without modifying core SCADA telemetry workflows.

1. SearchBar UI
Dropdown component inside App header. Captures keyboard shortcuts (Ctrl+K) and debounces inputs.
2. Search Brain (useSearch)
React hook managing search queries, active context filters, ranking, and race conditions.
3. Data Connectors
Modular adapters querying Logs, Parts, and Network Nodes via fuzzy text indexing.
03 // EVALUATION MATRICES

Alternatives Considered

During the design phase, we evaluated multiple implementation strategies to balance complexity, performance, and developer velocity:

The "All-in-One" Monolithic Function

Consolidate all searching instructions for logs, parts, and nodes into a single giant search handler block.

✓ Quick Start: Extremely easy to implement initially. Minimum file boilerplate.
✗ Poor Scalability: Adding a new domain in the future requires rewriting core logic, increasing the risk of introducing regressions.

Screen-Specific Search Boxes Only

Isolate search capabilities inside separate panels. An operator can only search logs on the Execution Console tab, and parts on the Inventory tab.

✓ Absolute Isolation: Zero component coupling. Each component manages its own data cleanly.
✗ Multi-click Friction: Operators must click away to a completely new screen just to execute a search, failing to preserve live context.

Heavy External Search Servers

Leverage elastic-level databases or heavy external frameworks to index the datasets.

✓ Future Proofing: Handles millions of datasets easily with pre-computed index caching.
✗ Unnecessary Bloat: Our client-side telemetry buffer operates on ~100 items, meaning simple in-memory Fuse.js search runs sub-millisecond.

05 // TECHNICAL SPECS & TYPES

Agreed Implementation Details

The following types and configurations were strictly defined to keep all components in complete alignment:

src/types.ts 
export interface SearchResult {
  id: string;
  title: string;
  subtitle?: string;
  type: 'log' | 'part' | 'node';
  action: {
    type: 'NAVIGATE' | 'OPEN_DRAWER' | 'HIGHLIGHT_MAP';
    payload: any; 
  };
}

Results trigger specific application actions:

  • NAVIGATE: Transitions active tabs (e.g. opens Inventory Screen).
  • OPEN_DRAWER: Toggles slide-over maintenance drawer with detailed telemetry history.
  • HIGHLIGHT_MAP: Recenters and flashes the target pipeline valve on the Topology map.
